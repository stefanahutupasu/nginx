NGINX Presentation and Installation Guide

What is Nginx? Nginx is a web server and reverse proxy server that was first released in 2004. It is free and open source software that can be used to serve static and dynamic web pages and applications. Nginx is designed to handle high levels of concurrent connections and requests, making it a popular choice for high-traffic websites and applications.

How is Nginx helpful and why someone might want or need to use it? There are several reasons why someone might want or need to use Nginx:

  1. High performance: Nginx is designed to handle a large number of simultaneous connections and requests, making it a great choice for websites and applications that expect high traffic.
  1. Reverse proxy: Nginx can be used as a reverse proxy, which means it can sit in front of web servers and forward requests to them. This can help to improve performance and security by offloading tasks such as SSL encryption and load balancing.
  1. Load balancing: Nginx can be used as a load balancer, distributing requests across multiple web servers to ensure that no single server becomes overloaded.
  1. Caching: Nginx can be used to cache frequently requested content, reducing the load on web servers and improving response times for users.
  1. Security: Nginx includes several security features such as SSL/TLS encryption, access control, and rate limiting, making it a popular choice for secure web applications.

Step-by-Step Installation Tutorial for Nginx:

  1. Update the package index on your server:
    sudo apt update
  1. Install Nginx:
    sudo apt install nginx
  1. Check that Nginx is running:
    sudo systemctl status nginx
  1. If Nginx is not running, start the service:
    sudo systemctl start nginx
  1. Check that Nginx is running again:
    sudo systemctl status nginx
  1. Open your web browser and navigate to your server's IP address. You should see the Nginx welcome page.

Nginx Configuration Files

NGINX's configuration files end with the .conf extension and usually live inside the /etc/nginx/directory.

Among these files, there should be one named nginx.conf. This is the the main configuration file for NGINX.

It is usually not recommended to edit this file unless one really knows what they’re doing. For learning purposes, renaming this file to be kept as a backup is recommended. After that, create a new one.

# renames the file
sudo mv nginx.conf nginx.conf.backup

# creates a new file
sudo touch nginx.conf

Validate and Reload Configuration Files

The first step after creating a new configuration file or changing an existing one is to check it for syntax errors. The -t option is available in the nginx binary for doing this.

sudo nginx -t

# nginx: the configuration file /etc/nginx/nginx.conf syntax is ok
# nginx: configuration file /etc/nginx/nginx.conf test is successful

The configuration file is fine, but NGINX isn't going to use it. NGINX operates by reading the configuration file only once and then continuing working based on that.

You must explicitly tell NGINX to reload the configuration file whenever you make changes to it.

One way is to restart NGINX:

sudo systemctl restart nginx

Directives and Contexts in NGINX

Technically, everything inside a NGINX configuration file is a directive. Directives are of two types:

  • Simple Directives
  • Block Directives

A simple directive consists of the directive name and any necessary parameters, such as listen, return, and others. Semicolons are used to end simple directives.

As opposed to simple directives, block directives end with a pair of curly braces that enclose extra instructions, rather than a semicolon.

We can see that in our config file, we have both types of directives.

A block directive capable of containing other directives inside it is called a context, that is events, http and so on. There are four core contexts in NGINX:

  • events { } – The events context is used for setting global configuration regarding how NGINX is going to handle requests on a general level. There can be only one events context in a valid configuration file.
  • http { } – Evident by the name, http context is used for defining configuration regarding how the server is going to handle HTTP and HTTPS requests, specifically. There can be only one http context in a valid configuration file.
  • server { } – The server context is nested inside the http context and used for configuring specific virtual servers within a single host. There can be multiple server contexts in a valid configuration file nested inside the http context. Each server context is considered a virtual host.
  • main – The main context is the configuration file itself. Anything written outside of the three previously mentioned contexts is on the main context.

So, a configuration file may contain multiple server contexts. But how does NGINX choose which of those contexts should handle a request when it comes to the server?
